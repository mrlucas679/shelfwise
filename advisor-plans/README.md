# ShelfWise Data-Truth Integration Plans

Generated by the `improve` audit on 2026-07-13 against commit `5e306c2` and the
dirty `developers` worktree. These plans are separate from `plans/`, whose five
existing plans describe earlier release work. Do not reset or discard the dirty
worktree before execution; it contains the digital-twin and production work this
audit evaluated.

The audit followed the complete path from producers through contracts, stores,
workers, cascades, tools, chat, HITL, learning, observability, the frontend, and
deployment tests. The present implementation separates training rows and live
chat in some places, but data-domain identity is not yet carried through the
whole application.

## Dependency Notes

- Plan 001 creates the one authoritative `data_domain` value and puts the twin
  guard inside `TwinService`, where every caller must pass.
- Plan 002 depends on that value to select generated-world or exact-store facts.
- Plan 003 depends on both so persisted decisions and side effects are partitioned
  by the domain that actually produced them.
- Plan 004 can begin after plan 001, but its final cascade tests require plan 002.
- Plan 005 is the release gate. It must test the behavior delivered by all prior
  plans rather than merely checking that endpoints return non-empty responses.

## Vetted Findings

1. `TwinService.project_event()` has no domain guard. The API helper guards one
   path, but event bootstrap and the reachable `TwinProjectionWorker` call the
   service directly. A replay can therefore import generated events into reported
   operational state.
2. Both the direct cascade dispatcher and production queue worker run operational
   events against `WorldFactsProvider`. Product endpoints and the platform-tool
   catalogue also default to generated-world facts.
3. Decisions, learning thresholds, open orders, candidates, write-back tasks,
   traces, and model runs are tenant-scoped but not data-domain-scoped. A drill can
   change the same threshold or operational suppression state used by a real shop.
4. The threshold tools call `memory.thresholds()` without `tenant_id`, despite the
   stores supporting tenant filters.
5. Cold-chain event projection drops `temp_c`; live stock accepts a SKU but does
   not filter by it; live cold-chain queries return only diagnosis by default; and
   `live_context()` truncates before relevance filtering.
6. Scenario properties persist but scenario branch metadata is process-local, so
   comparison breaks after a restart.
7. The frontend approval queue has no `data_domain`, and product copy describes
   generated-world data as a live query.
8. Agentic chat creates a fresh `AuditLog` per request, so `/tools/platform/audit`
   does not contain the chat tool calls it claims to expose.
9. Production CI asks generated-world questions while production chat defaults to
   the operational twin. It checks only latency/non-empty text, not provenance,
   selected tools, state isolation, or data-domain correctness.

## Findings Deferred

- The `TwinProjectionWorker` is instantiated but is not currently started by the
  application lifespan. Plan 001 still guards it because it is a valid exported
  execution path and would become unsafe as soon as it is enabled.
- Conversation identity is not a current blocker. The chat store already keys
  state and locking by `(tenant_id, user_id, conversation_id)` and supports
  `message_id` replay checks.
- Adding Playwright is desirable, but it is a new dependency. Plan 005 requires
  explicit operator approval before adding it, in line with repository rules.

## Global Stop Conditions

- Do not execute these plans on `main`; use `developers`.
- Do not reset, clean, stash, or overwrite unrelated dirty work.
- Do not use heuristics such as ID prefixes as the steady-state domain contract.
  Heuristics are permitted only in a one-time migration for legacy rows.
- Do not make operational mode silently fall back to generated-world facts.
- Do not allow a simulation approval to create an operational write-back task.
- Stop if an implementation requires live GPU credit. Plans 001-004 and the local
  portions of plan 005 must pass without starting the cloud droplet.
